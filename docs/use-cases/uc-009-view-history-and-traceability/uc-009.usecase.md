## Use Case Description

## User story
As an authorized staff member, I want to view the history and traceability of handover information and Resident-related records, so I can understand what changed, when, and by whom.

### Brief Use Case
**Title**: View history and traceability
**Success Flow**:
The authorized user opens the history view for a selected record (e.g., a handover entry), and the system displays a chronological audit trail with timestamps, authors, and change details, including clear marking of late/retroactive entries.

### Casual Use Case
**Title**: View history and traceability
**Scope**: Slottets Drifttavlen (Audit trail)
**Level**: User Goal
**Actors**:
- Authorized Staff Member
- System

**Main Flow**:
1: Authorized Staff Member opens the history/traceability view.
2: System displays a list of auditable record types and/or recent records.
3: Authorized Staff Member selects a record and optionally chooses a time range.
4: System shows the record history as a chronological list of changes (who, when, what).
5: Authorized Staff Member selects a history entry.
6: System displays change details and clearly marks late/retroactive entries with their original and registered timestamps.
7: Authorized Staff Member closes the history view.

**Main Extensions**:
1a: User is not authorized.
    1: System denies access and shows an authorization error.
2a: History data cannot be loaded.
    1: System shows an error and offers retry.
4a: No history exists for the selected record/time range.
    1: System shows a "no records" message.
6a: Change details cannot be displayed.
    1: System shows an error and returns to the history list.

**Summary**: Authorized users can see who changed what and when, supporting accountability and safe care decisions.

### Fully Dressed Use Case
**Title**: View history and traceability
**Scope**: Slottets Drifttavlen (Audit trail)
**Level**: User Goal
**Actors**:
- Authorized Staff Member
- System

**Related Requirements**:
- [REQ-F-006] History and traceability for all handovers and changes (audit trail).
- [REQ-F-007] Marking of late/retroactive entries with correct timestamp.
- [REQ-R-003] Audit log for all changes and user actions.

**Preconditions**:
- System is operational.
- User is logged into the system.
- User has permission to view history/audit trail.
- At least one auditable record exists (or the system can still show an empty history state).

**Main Flow**:
1. Authorized Staff Member opens the history/traceability view.
2. System requests the history for the selected record.
3. System displays a chronological list of changes, including who performed the action and when.
4. Authorized Staff Member selects a change entry.
5. System displays details of the change (what was changed and previous value where available).
6. System clearly marks late/retroactive entries and shows both the original event time and the registration time.

**Extensions**:
    a: At any time, if an error occurs, system shows an error and offers retry.
    b: At any time, if the user is not authorized, system denies access and logs the attempt.
    2a: History cannot be retrieved (e.g., database connection error).
        1: System shows an error and offers retry.
    3a: The selected record has no history.
        1: System shows a "no records" message.
    5a: Change details are unavailable (e.g., missing referenced data).
        1: System shows an error and returns to the list.

**Postconditions**:
- The user has viewed the available history for the selected record.
- No business data is modified by viewing history.
- The system supports accountability by presenting who/when/what for changes.
