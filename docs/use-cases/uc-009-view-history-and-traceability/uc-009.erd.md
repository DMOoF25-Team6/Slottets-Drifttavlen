## ERD Diagram
```mermaid
erDiagram
    RESIDENT {
        PK Identifier string
        Initials string
    }
    USER {
        PK Identifier string
    }
    OVERLAP {
        PK Id guid
        Start datetime
        End datetime
        Shift string
        Department string
        Summary string
        FK ChangeHistoryId guid
    }
    CHANGEHISTORY {
        PK Id guid
        From datetime
        To datetime
    }
    AUDITLOG {
        PK Id guid
    }
    AUDITENTRY {
        PK Id guid
        EventTime datetime
        RegisteredTime datetime
        Entity string
        ChangeType string
        Description string
        FK AuditLogId guid
        FK UserId string
        FK ResidentId string
        FK OverlapId guid
    }
    CHANGEDETAIL {
        PK Id guid
        Field string
        OldValue string
        NewValue string
        FK AuditEntryId guid
    }

    RESIDENT ||--o{ AUDITENTRY : relatesTo
    USER ||--o{ AUDITENTRY : changedBy
    OVERLAP ||--o{ AUDITENTRY : relatesTo
    CHANGEHISTORY ||--o{ OVERLAP : has
    AUDITLOG ||--o{ AUDITENTRY : stores
    CHANGEHISTORY ||--o{ AUDITENTRY : contains
    AUDITENTRY ||--o{ CHANGEDETAIL : details
```

## Assumptions and Dependencies
- All auditable changes and relevant user actions create immutable `AUDITENTRY` records in the `AUDITLOG`.
- Late/retroactive entries are represented by both `EventTime` (when it happened) and `RegisteredTime` (when it was entered).
- `CHANGEHISTORY` is a filtered view (by record/time range) of `AUDITENTRY` records presented to the user.
- Access to audit/history requires authorization; unauthorized access attempts are logged.

## Notes
- The UC-009 history and traceability view is read-only; it does not change business data.
- The UI presents audit information as a chronological list and provides details per selected entry.
- `Entity` in `AUDITENTRY` is shown to users as a recognizable record type (e.g., ResidentNote, Task, PhoneAssignment, Overlap).
